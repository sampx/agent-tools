# 技能评估 (Evaluate Skill)

## 目标

对指定技能进行全面评估，生成标准化报告，帮助决策是否安装、备份或删除。

## 待评估技能

目标路径: `$ARGUMENTS`

> **注意**: 如果没有提供参数，询问用户。

---

## 评估流程

### 1. 信息收集

对每个技能执行：

```bash
# 检查目录结构
ls -la <skill-path>/

# 统计文件数量和类型
find <skill-path> -type f | wc -l

# 统计 SKILL.md 行数
wc -l <skill-path>/SKILL.md
```

阅读核心文件：
- `SKILL.md` - 主要技能定义（必读）
- `scripts/` - 辅助脚本（如有）
- `references/` - 参考资料（如有）
- `templates/` - 模板文件（如有）

### 2. 功能分析

提取以下信息：
- **名称**: SKILL.md 头部 metadata 的 `name`
- **描述**: SKILL.md 头部 metadata 的 `description`
- **核心功能**: 技能提供的主要能力
- **触发场景**: 什么情况下应该使用此技能
- **依赖工具**: 需要的外部工具或 API

### 3. 质量评估

按以下维度评分（1-5 星）：

| 维度 | 评分标准 |
|------|----------|
| 内容完整度 | SKILL.md 内容深度、示例丰富度、边界覆盖 |
| 实用性 | 解决实际问题的能力、使用频率预期 |
| 可执行性 | 是否有脚本辅助、流程是否清晰可操作 |
| 文档质量 | 结构清晰度、格式规范、易于理解 |
| 维护成本 | 依赖稳定性、更新频率需求 |

### 4. 关系分析

- **与已安装技能的关系**: 功能重叠、互补、独立
- **与项目需求匹配度**: 当前项目是否会用到
- **替代方案**: 是否有更优选择

**触发条件**: 路径包含 `.agents`, `.claude`, `.opencode`, `openclaw`, `universal` 或 `backup`。
**报告目标**: 忽略“去留决策”，着眼于**透彻解析技能的机制、使用场景和操作方法**。

```markdown

- **场景 B**: {具体触发场景描述}

### 3. 工作原理与机制 (Under the Hood)
- **依赖工具**: {如使用哪些外部 CLI、脚本、API}
```

## 评分标准详解

### ⭐⭐⭐⭐⭐ (5星 - 优秀)
- 内容完整、示例丰富
- 有脚本/工具支持
- 解决高频痛点
- 文档结构清晰

### ⭐⭐⭐⭐ (4星 - 良好)
- 内容较完整
- 流程清晰可执行
- 有一定实用价值
- 文档规范

### ⭐⭐⭐ (3星 - 一般)
- 内容基本完整
- 纯文档无脚本
- 使用场景有限
- 文档可读

### ⭐⭐ (2星 - 较差)
- 内容较浅
- 缺乏实际指导
- 使用场景模糊
- 文档不完整

### ⭐ (1星 - 很差)
- 仅有框架/空壳
- 无实际内容
- 不具备使用价值

---

## 决策框架

### 建议安装 (✅)
- 综合评分 ≥ 4 星
- 与项目需求匹配
- 无功能重叠或互补明显
- 维护成本低

### 备份待定 (⚠️)
- 综合评分 3 星
- 可能有未来需求
- 与现有技能有部分重叠
- 作为知识储备

### 建议删除 (❌)
- 综合评分 ≤ 2 星
- 内容不完整或为空壳
- 与其他技能高度重叠
- 无实际使用价值

## 执行建议

评估完成后，询问用户下一步操作（根据评估的类型选择）：

**对于 INBOX 技能 (入库评估):**
```
评估完成。是否执行以下操作？
1. 安装推荐技能: /skill-deployer
2. 删除低价值技能
3. 导出评估报告到文件
```

**对于已安装/Backup技能 (深度解析):**
```
深度解析完成。是否执行以下操作？
1. 立即在此上下文中测试该技能用法
2. 根据原理解析，优化或调优此技能代码
3. 导出深度解析报告到文档
```

---

## 注意事项

- 评估应客观中肯，避免主观偏见
- 对于不熟悉的领域，标注不确定性
- 优先考虑项目实际需求而非技能数量
- 重叠技能建议保留质量更高的一个
