# 创建 PRD: 新产品需求文档

## 使用方式

```bash
/create-prd                    # 交互式输入产品名称
/create-prd my-new-product     # 指定产品名称
```

## 定位

**仅用于新产品**。现有项目的新功能请使用 `/plan-feature`。

## 流程

### 1. 确定产品名称

- 从参数获取，或询问用户
- 输出路径：`docs/products/PRD-{product-name}.md`

### 2. 收集上下文

**必读：**
- `MEMORY.md` - 获取工作空间架构经验
- `.workspace.md` - 了解工作空间的结构和设计

**可选：**
- 如果有参考资料的讨论历史，从中提取需求

### 3. 生成 PRD

基于对话上下文生成完整 PRD，使用以下模板结构：

---

## PRD 模板结构

### 1. 执行摘要 (Executive Summary) [必选]

简洁的产品概述（2-3 段），包含核心价值主张和 MVP 目标陈述。

### 2. 使命 (Mission) [必选]

**使命声明**: 一句话描述产品使命

**核心原则**: 3-5 条关键原则

### 3. 目标用户 (Target Users) [必选]

| 用户类型 | 技术水平 | 核心需求 | 痛点 |
|---------|---------|---------|------|
| [用户 A] | [低/中/高] | [需求] | [痛点] |

### 4. MVP 范围 (MVP Scope) [必选]

**范围内 (In Scope)**:
- 核心功能、技术、集成、部署要求（使用 checkbox）

**范围外 (Out of Scope)**:
- 延后到后续阶段的功能（使用 checkbox）

### 5. 用户故事 (User Stories) [必选]

格式：`As a [user], I want to [action], so that [benefit]`

每个故事包含验收标准和示例，5-8 个主要故事。

### 6. 核心架构与模式 (Core Architecture & Patterns) [可选]

- 高层架构方案
- 目录结构
- 关键设计模式

### 7. 工具/功能 (Tools/Features) [可选]

详细的功能规格，包含用途、操作和关键特性。

### 8. 技术栈 (Technology Stack) [必选]

| 类别 | 技术 | 版本 |
|------|------|------|
| 后端 | [技术] | [版本] |
| 前端 | [技术] | [版本] |

包含核心依赖、可选依赖、第三方集成。

### 9. 安全与配置 (Security & Configuration) [可选]

认证方式、配置管理、安全范围、部署考虑。

### 10. API 规格说明 (API Specification) [仅 API 项目]

端点定义、请求/响应格式、认证要求。

### 11. 成功标准 (Success Criteria) [必选]

- MVP 成功定义
- 功能需求（checkbox）
- 质量指标
- 用户体验目标

### 12. 实现阶段 (Implementation Phases) [必选]

拆分为 3-4 个阶段，每阶段包含目标、交付物、验证标准。

### 13. 未来考虑 (Future Considerations) [可选]

MVP 后的增强功能、集成机会、高级功能。

### 14. 风险与缓解 (Risks & Mitigations) [可选]

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|

### 15. 附录 (Appendix) [可选]

相关文档、关键依赖、仓库结构。

---

## 简化规则

小型项目可简化，聚焦核心章节：**1-5, 8, 11, 12**

标注因信息缺失而做出的假设。

### 4. 质量检查

- [ ] 所有必选章节已填写
- [ ] MVP 范围现实且定义清晰
- [ ] 技术选型有合理依据

### 5. 输出确认

1. 确认文件路径：`docs/products/PRD-{product-name}.md`
2. 简要总结 PRD 内容
3. 建议后续：评审后可使用 `/plan-feature` 创建实现计划
